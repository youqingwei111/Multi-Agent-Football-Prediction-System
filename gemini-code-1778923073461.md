# 欧洲足球赛事多 Agent 智能分析系统开发文档 (MVP版)

## 1. 项目描述

本项目旨在开发一个基于大语言模型（LLM）的多 Agent 协作系统，模拟专业的足球赛事分析团队。系统通过自动化流转，完成从外部数据抓取、深度战术推演到最终报告排版的完整业务闭环。

针对实习求职与真实工程场景，本项目拒绝“过度设计”与不可控的黑盒编排，核心聚焦于**系统鲁棒性**与**复杂业务流转**。通过引入状态机模型控制 Agent 行为，重点攻克大模型开发中的“工具调用（Tool Calling）失败重试”与“数据幻觉降级”等核心工程痛点，最终交付一个逻辑严密、具备极客感且真实可用的最小可行性产品（MVP）。

系统核心角色设计：

- **情报球探 (Scout Node)：** 负责与外部体育 API 交互，精准抓取赛事基本面、伤停名单数据。
- **战术分析师 (Analyst Node)：** 接收球探数据，利用大模型推理能力进行战术克制与胜率推演。
- **报告主编 (Editor Node)：** 剔除冗余信息，将推演草稿标准化为结构清晰的 Markdown 分析报告。

---

## 2. 所用技术栈

本项目采用前后端分离架构，核心逻辑处理全面拥抱 Python AI 生态：

### AI 与后端核心 (Python 生态)

- **Agent 编排框架：** **LangGraph**。利用其图结构（Graph）与全局状态（State）管理能力，实现极具容错性的多 Agent 循环工作流编排。
- **Web 框架：** **FastAPI**。提供高性能的异步 RESTful/SSE 接口，完美适配大模型流式输出与前端实时交互需求。
- **数据持久层：** **PostgreSQL** + **SQLModel**。负责存储赛事基础数据、Agent 运行快照（Checkpoints）及最终生成的分析报告。
- **大模型基座 (LLM)：** DeepSeek / 通义千问 / OpenAI (优先选择支持高精度 Tool Calling 的模型)。

### 前端展示与外部交互

- **前端框架：** **Vue3** + Vite。
- **界面 UI：** TailwindCSS (或类似轻量级组件库)，打造极简“控制台/黑客帝国”风格的实时日志流交互界面。
- **外部数据源：** API-Football (或同等提供历史战绩、伤病与实时赔率的体育 API)。

## 3. 开发路径 (4周敏捷冲刺)

项目开发遵循“自底向上、先稳后连”的原则，拆分为四个阶段：

### 阶段一：底层基建与状态定义 (第 1-7 天)

**核心目标：跑通外部真实数据，定义全局流转字典。**

1.  **环境初始化：** 搭建 Python 虚拟环境，集成 FastAPI 与基础依赖。
2.  **数据源打通：** 注册获取体育 API 密钥，编写原生 Python 脚本（不接入 AI），成功抓取指定比赛的 JSON 数据（赔率、近期战绩等）。
3.  **定义 State：** 在 LangGraph 中使用 `TypedDict` 定义 `MatchAnalysisState`，明确“对阵双方、情报原始数据、分析草稿、重试计数器”等全局状态字段。

### 阶段二：节点开发与单点联调 (第 8-14 天)

**核心目标：完成各个 Agent 的核心逻辑封装，攻克 Tool Calling。**

1.  **开发球探节点 (Scout)：** 将第一周的 API 脚本封装为工具。测试大模型提取参数并触发网络请求的能力，将获取到的 JSON 写入 State。
2.  **开发分析与主编节点 (Analyst & Editor)：** 编写高质量 Prompt，测试从 State 读取数据、进行推演并标准化输出 Markdown 的逻辑。
3.  **鲁棒性建设：** 针对球探节点，编写异常捕获逻辑（如网络超时、JSON 解析失败），为下一步的循环图做准备。

### 阶段三：LangGraph 图编排与数据入库 (第 15-21 天)

**核心目标：将散落的节点串联为具备容错能力的状态机。**

1.  **构建 StateGraph：** 将 Scout、Analyst、Editor 三个节点按业务顺序相连。
2.  **配置 Conditional Edges (条件边)：** 在核心节点后加入验证逻辑。例如：若 Scout 节点失败，触发回流并增加 State 中的错误计数；若重试 3 次仍失败，则走降级节点，抛出优雅的异常。
3.  **持久化 (Checkpointer)：** 引入 PostgreSQL 与 SQLModel，利用 LangGraph 的持久化机制，将每一轮的流转状态和最终报告存入数据库。

### 阶段四：全栈联调与前端呈现 (第 22-28 天)

**核心目标：打破黑盒，实现可视化的终端交互产品。**

1.  **FastAPI 接口封装：** 将编译完成的 LangGraph 包装为可通过 HTTP 请求触发的接口，并支持日志的流式传输 (Streaming)。
2.  **Vue3 控制台开发：** 构建左右分栏界面。左侧提供赛事输入入口，中间展示类似终端打字的 Agent 实时状态流转日志，右侧用于渲染最终的 Markdown 报告。
3.  **边界联调与优化：** 测试错误输入、API 熔断等极端情况，优化 Prompt 剔除“机器味”废话。录制完整的项目演示视频。
